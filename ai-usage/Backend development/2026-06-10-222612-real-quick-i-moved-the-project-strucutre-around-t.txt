● Great question—this is exactly the kind of tradeoff thinking that matters in
  production systems.

  Performance with 40 iterations:
  On a modern processor (i7/Ryzen 7), your current settings would take roughly
  2-5 seconds per hash operation. That's slow for user registration/password
  changes.

  Security tradeoff by iterations:

  ┌────────────┬────────────┬──────────────────────────────────────┐
  │ Iterations │    Time    │ Security Against Offline Brute Force │
  ├────────────┼────────────┼──────────────────────────────────────┤
  │ 2-3        │ ~200-400ms │ ✓ Good (OWASP recommendation)        │
  ├────────────┼────────────┼──────────────────────────────────────┤
  │ 10         │ ~500ms-1s  │ ✓✓ Strong                            │
  ├────────────┼────────────┼──────────────────────────────────────┤
  │ 40         │ 2-5s       │ ✓✓✓ Maximum                          │
  └────────────┴────────────┴──────────────────────────────────────┘

  What iterations actually protect against:
  - They slow down offline attacks (if someone steals your password database)
  - They don't directly help with online attacks (rate limiting is what stops
  those)

  Real-world context:
  - OWASP recommends Iterations: 2-3 for Argon2
  - Microsoft Identity uses lower iterations by design
  - The bigger security wins come from:
    a. Rate limiting (lock after 5 failed attempts) ← more important
    b. Strong password requirements
    c. 2FA (which you're planning)

  My take:
  I'd recommend dropping to Iterations: 3 for your project:
  - ~300-400ms per hash (responsive UX)
  - Still shows security consciousness
  - Still way better than plain text passwords
  - Still very hard to crack offline

  But if you want to showcase maximum security focus in the interview, keep 40.
  It's not wrong, just slower. Your call!

  Basic usage pattern:
  // Success result
  public async Task<Result<Guid>> CreateFolderAsync(CreateFolderCommand cmd)
  {
      if (string.IsNullOrEmpty(cmd.FolderName))
          return Result.Fail("Folder name is required");

      var folder = new FolderEntity { ... };
      _context.Folders.Add(folder);
      await _context.SaveChangesAsync();

      return Result.Ok(folder.Id);
  }

  // In handler/controller
  var result = await _service.CreateFolderAsync(command);
  if (result.IsSuccess)
      return Ok(result.Value); // The Guid
  else
      return BadRequest(result.Errors); // Array of errors

  Benefits:
  - ✅ No exceptions for expected errors (validation, not found, etc.)
  - ✅ Easy to chain and compose operations
  - ✅ Clear success/failure semantics
  - ✅ Multiple errors in one result
  - ✅ Cleaner than Try/Catch patterns
  - ✅ Better for CQRS handlers
